# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 概要

`KyoshinMonitorLib` は、防災科研(NIED)の「強震モニタ」を .NET から扱うためのライブラリ群です。NuGet で配布される複数パッケージで構成されています。コメント・XML ドキュメントは日本語で書かれているので、追記する場合も日本語に揃えてください。

## ビルド・実行

```pwsh
# ソリューション全体をビルド (各 csproj は GeneratePackageOnBuild=true なので Release で .nupkg も生成される)
dotnet build KyoshinMonitorLib.sln
dotnet build KyoshinMonitorLib.sln -c Release

# 単一プロジェクトのビルド
dotnet build src/KyoshinMonitorLib/KyoshinMonitorLib.csproj
```

### テストについて (重要)

`src/Tests` は **単体テストではなく、実際の強震モニタ API を叩くライブ統合テスト用のコンソールアプリ** (トップレベルステートメント) です。xUnit/NUnit 等のテストフレームワークは導入されていません。

```pwsh
dotnet run --project src/Tests
```

実行には以下が必要です:
- カレントディレクトリに観測点情報ファイル `ShindoObsPoints.mpk.lz4` (lz4 圧縮済み mpk) が存在すること
- インターネット接続 (本物の `bosai.go.jp` / NTP サーバーへアクセスする)

機能を検証したいときは `Program.cs` を編集して動作確認する、というのがこのリポジトリの流儀です。

### ベンチマーク (`src/Benchmark`)

画像からの色取得 (`ParseScaleFromImage`) の性能計測用の **BenchmarkDotNet** コンソールアプリ。`net10.0` 単独・`IsPackable=false`。ネットワークや外部ファイルに依存せず、合成 GIF (352x400・パレットカラー) と約 1300 点のダミー観測点を生成して計測する。

```pwsh
dotnet run --project src/Benchmark -c Release            # ベンチマーク実行
dotnet run --project src/Benchmark -c Release -- verify  # 最適化版が現行実装と同じ結果になるか検証
```

- 最適化前後を A/B 比較できるよう、最適化前の実装 (Legacy) を `Legacy.cs` に自己完結で複製している (ライブラリ本体を変えても比較基準がぶれないようにするため)。色変換も複製する。
- BenchmarkDotNet は実行時に `bin` 配下へ Autogenerated プロジェクトを生成しビルドする。これがルートの `Directory.Build.props` (複数 TargetFrameworks) を継承するとビルドに失敗するため、`src/Benchmark/Directory.Build.props` を置いてサブツリーをルート props から遮断している (親を import しない)。

## マルチターゲットとバージョン (一元管理)

**共通のビルド設定・バージョン・ターゲットフレームワークはリポジトリルートの `Directory.Build.props` で一元管理**しています。バージョン番号やターゲットを変えるときは原則ここ1か所を編集すれば全ライブラリに反映されます。各 `.csproj` にはプロジェクト固有の設定 (PackageReference, ProjectReference 等) のみ残してあります。

- ライブラリ共通ターゲット: `net10.0;net8.0;netstandard2.1;netstandard2.0;net47` (`Directory.Build.props` の `<TargetFrameworks>`)
- バージョン: `Directory.Build.props` の `<Version>` (全パッケージ共通)
- Tests: `net10.0` 単独。`Directory.Build.props` の `TargetFrameworks` を `<TargetFrameworks></TargetFrameworks>` で打ち消し、`<TargetFramework>net10.0</TargetFramework>` を指定。`<IsPackable>false</IsPackable>` でパッケージ化を抑止している。
- サポート終了 (EOL) した .NET (net5.0/net6.0/net7.0) はターゲットから除外済み。モダン .NET は LTS の net10.0 / net8.0 のみ。`netstandard2.0`/`netstandard2.1`/`net47` は互換ターゲットとして維持。
- ターゲット別の条件コンパイル (`#if NET...`) は使っていないため、全ターゲットで同一コードがコンパイルされる。`netstandard2.0`/`net47` も対象なので、新しい言語/BCL API を使う際は古いターゲットでコンパイルが通るか注意 (`LangVersion=latest`, `Nullable=enable`)。
- 依存パッケージ `System.Text.Json` (参照するのはコアのみ) は、ビルドするターゲットのランタイムに合わせてバージョンを切り替える。`KyoshinMonitorLib.csproj` の `$(SystemTextJsonVersion)` プロパティで、net10.0 は `10.0.*`、それ以外 (net8.0/netstandard/net47) は `8.0.*` のフロート指定にしており、各メジャーの最新パッチを自動取得する。
- 既知の警告: `Images` プロジェクトで `System.Drawing` 利用箇所に **CA1416** (Windows 専用 API) が多数出るが、これは Windows 前提の既存警告でビルドは通る。

## アーキテクチャ

### パッケージ構成と依存関係

```
KyoshinMonitorLib            ← コア (API クライアント, モデル, 観測点 I/O, NTP)
├── KyoshinMonitorLib.Images       ← System.Drawing による画像解析 (拡張メソッド)
├── KyoshinMonitorLib.SkiaImages   ← SkiaSharp による画像解析 (拡張メソッド)
└── KyoshinMonitorLib.Timers       ← SecondBasedTimer (コアに非依存・独立)
```

### API クライアント層 (`src/KyoshinMonitorLib`)

- `Api` (抽象基底, `Api.cs`): `HttpClient` を 1 つ保持し、`GetJsonObject<T>()` / `GetBytes()` を提供。`IDisposable`。タイムアウト時は `KyoshinMonitorException` を投げる。
- 派生クライアントは 3 種:
  - `WebApi` — Web 版強震モニタ (`lmoni` / `smi.lmoniexp` ホスト)
  - `LpgmWebApi` — 長周期地震動モニタ。`WebApi` のエンドポイントに加え長周期地震動系の画像も取得可能。**新規開発では基本こちらを使う** (Tests もこちら)。
  - `AppApi` — スマホアプリ API。**ほとんどのエンドポイントが現在サーバー側で利用不可**。観測点情報との結合 (`GetLinkedRealtimeData`) とキャッシュ処理を持つ。
- **URL 生成は API クラスから分離**されている (`UrlGenerator/` 配下)。`WebApiUrlGenerator` / `LpgmWebApiUrlGenerator` / `AppApiUrlGenerator` が静的に bosai.go.jp の URL テンプレートを保持し、API クラスがそれをラップする。エンドポイント URL を変えるときはここを編集する。
- `RealtimeDataType` enum (`UrlGenerator/RealTimeDataType.cs`) が取得データ種別 (震度/PGA/PGV/PGD/速度応答/長周期階級) を表し、`ToUrlString()` で URL 用文字列に変換する。長周期 (`Abrsp*`) 系は Lpgm 系列でのみ有効。
- 返り値は常に `ApiResult<T>` (`StatusCode` + `Data`)。失敗時 `Data` は null。EEW 系は未発表時に HTTP 404 が返る。

### JSON と AOT 対応

JSON デシリアライズは **System.Text.Json のソースジェネレーター** を使用。`ApiResult/ApiResultJsonContext.cs` の `[JsonSerializable]` 属性に型 (`Eew`, `RealtimeData`, `SiteList`) を登録し、`Api.GetJsonObject` には対応する `JsonTypeInfo` を渡す。**新しい API レスポンス型を追加したらこの Context にも登録する**こと (リフレクションベースの汎用シリアライズは使わない設計)。

### 画像解析層 (Images / SkiaImages)

`Images` と `SkiaImages` は **同一の機能を別の画像ライブラリで実装した並行実装**です。利用側はプラットフォームに応じてどちらか一方を選ぶ:
- `Images`: `System.Drawing.Common` ベース。`AllowUnsafeBlocks` で `LockBits` + `Span<byte>` を使い高速にピクセル走査。
- `SkiaImages`: `SkiaSharp` ベース。クロスプラットフォーム。`SKBitmap.GetPixel` を使用。

両者は `ColorConverter` / `ImageAnalysisResult` / `Extensions` (`ParseScaleFromParameterAsync`, `ParseScaleFromImage`) を **それぞれ重複して持つ**。色変換アルゴリズム等を直すときは **両プロジェクトに同じ変更を入れる**必要がある。

解析フロー: `WebApi`/`LpgmWebApi` の拡張メソッド `ParseScaleFromParameterAsync(points, datetime, dataType)` → 画像取得 → 各観測点の `Point` 座標のピクセル色を取得 → `ColorConverter.ConvertToScaleAtPolynomialInterpolation()` で色→スケール(0〜1)に変換。
- スケールから震度/PGA/PGV/PGD への変換は `ImageAnalysisResult.GetResultToIntensity()` 等、または `ColorConverter.ConvertToXxxFromScale()`。**取得した画像種別 (`RealtimeDataType`) に応じて正しい変換メソッドを使い分ける**こと。
- 色→スケール変換は [JQuake の記事](https://qiita.com/NoneType1/items/a4d2cf932e20b56ca444) の多項式補間アルゴリズム (`ConvertToScaleAtPolynomialInterpolation`) が現行。テーブル参照版 `ConvertToIntensityAtTable` はレガシー。

### 観測点情報 (`ObservationPoint.cs`)

mpk (MessagePack, lz4 圧縮オプション付き) / JSON / CSV の読み書きをサポート。[KyoshinShindoPlaceEditor](https://github.com/ingen084/KyoshinShindoPlaceEditor) と互換。MessagePack の `[Key(n)]` / `DataContract` の `Order` はフォーマット互換のため変更しないこと。**lz4 圧縮済み mpk を非圧縮として読み込まない**よう `useLz4` 引数に注意。

### タイマー (`KyoshinMonitorLib.Timers`)

`SecondBasedTimer`: 強震モニタ取得用の 1 秒固定・ドリフト補正付きタイマー。`Stopwatch` ベースで誤差を蓄積させない。時刻補正 (`UpdateTime`) は自動では行われないので、別途 `NtpAssistance` 等で取得した時刻を定期的に流し込む運用。`Offset` でイベント発火を後ろにずらす (API 反映待ちのため通常 1〜2.5 秒)。
